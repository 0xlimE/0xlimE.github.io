Exploiting a stack overflow in a custom protocol parser

12 May 2026 · 8 min read · security · binary exploitation

A while back I was poking at a small IoT device and noticed something strange: it advertised a custom over-the-wire protocol on TCP port 9000. The first byte was a length prefix, the second a command tag, and the rest was payload. Predictable, but the parser had a subtle bug that turned out to be exploitable.

This post walks through how I found the vulnerability, built a working exploit, and — because the device shipped with NX but no ASLR — pivoted from a single-byte write to full RCE.

The protocol

Each packet starts with a single byte length field, followed by a one-byte command code and up to 253 bytes of payload. The server reads n + 2 bytes and dispatches based on the command code. Here's the relevant snippet I recovered from the firmware: 

void handle_packet(int sock) {
  char buf[256];
  uint8_t len;

  // read 1 byte length prefix
  recv(sock, &len, 1, 0);

  // then read len bytes of payload — what could go wrong?
  recv(sock, buf, len + 2, 0);

  dispatch(buf[0], buf + 1, len);
}

The bug is the len + 2. Because len is a uint8_t, a value of 0xFE reads exactly 256 bytes into a 256-byte buffer (fine), but 0xFF causes a one-byte overflow past the end of the stack frame. That single byte lands right on the saved frame pointer.

Setting up the exploit

I had three things going for me:

  • No stack canaries on this binary
  • NX was enabled, so shellcode on the stack was out
  • ASLR was disabled, which made ROP gadget addresses stable

First step was to find a useful gadget chain. I used ROPgadget to dump candidates, then filtered for anything that touched r0 and pc together.

A single byte overflow doesn't sound like much, but if you can nudge a saved frame pointer into a region you control, you effectively gain a stack pivot — and from there, everything is fair game.

Pivoting with one byte

The trick was getting the device to allocate a second buffer at a predictable address, then overwriting the low byte of the saved frame pointer to land inside it. The full exploit was about 40 lines of Python: 

import socket, struct

s = socket.socket()
s.connect((target, 9000))

# prime the heap with our ROP chain
s.send(build_priming_packet(chain))

# trigger the one-byte overflow
payload = b"A" * 255 + b"\x80"
s.send(bytes([0xff, 0x01]) + payload)

And that was enough to drop into a shell. The full PoC and a writeup of the gadget chain are in the repo linked at the top.
